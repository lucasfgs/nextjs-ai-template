## Story coverage standards

<Checklist
  items={[
    <>
      Show the <strong>default state</strong> and all user-facing variants that are expected in app
      code.
    </>,
    <>
      Capture at least one <strong>realistic composition</strong> so the component is documented in
      context, not only in isolation.
    </>,
    <>
      Include key <strong>interaction states</strong> such as disabled, checked, selected, loading,
      destructive, or empty states when relevant.
    </>,
    <>
      Add <strong>accessibility notes</strong> whenever a component needs labels, helper text,
      keyboard expectations, or visible focus behavior.
    </>,
    <>
      Keep stories <strong>co-located</strong> with the component file so implementation and
      documentation evolve together.
    </>,
  ]}

## When to add a new primitive

Add a new shared primitive only when the UI pattern is reused across features and the existing
library cannot express it with composition. If a need is isolated to one module or one screen, start
with local composition before growing the shared design-system surface.
